The Rust Programming Language

将所有内容整合在一起：Future、任务和线程

正如我们在第16章中看到的，线程提供了一种实现并发的方法。在本章中，我们看到了另一种方法：使用异步与未来和流。如果你在想何时选择一种方法而不是另一种，答案是：这取决于具体情况！在许多情况下，选择不是线程或异步，而是线程和异步。

许多操作系统几十年来一直提供基于线程的并发模型，许多编程语言也因此支持它们。然而，这些模型并非没有权衡。在许多操作系统中，每个线程都会占用相当多的内存。线程也仅在操作系统和硬件支持时才是一种选择。与主流的台式机和移动计算机不同，一些嵌入式系统根本没有操作系统，因此也没有线程。

异步模型提供了一组不同且最终互补的权衡。在异步模型中，并发操作不需要自己的线程。相反，它们可以在任务上运行，就像我们在流部分中使用trpl::spawn_task从同步函数启动工作一样。任务类似于线程，但不是由操作系统管理，而是由库级别的代码：运行时管理。

线程和任务的API如此相似是有原因的。线程作为一组同步操作的边界；并发可以在线程之间实现。任务作为一组异步操作的边界；由于任务可以在其主体中切换不同的未来（futures），因此并发可以在任务之间和任务内部实现。最后，未来（futures）是Rust中最细粒度的并发单元，每个未来可能代表其他未来的树。运行时——具体来说，其执行器——管理任务，而任务管理未来。在这方面，任务类似于轻量级的、由运行时管理的线程，这些线程具有由运行时管理而非操作系统管理所带来的额外功能。

这并不意味着异步任务总是优于线程（反之亦然）。在某些方面，使用线程的并发比使用async的并发更简单。这可以是优势也可以是劣势。线程 somewhat “fire and forget”；它们没有未来（future）的原生等效物，因此除了被操作系统本身中断外，它们会一直运行到完成。

事实证明，线程和任务通常配合得很好，因为任务可以在某些运行时中在不同线程之间移动。实际上，在底层，我们一直在使用的运行时（包括 spawn_blocking 和 spawn_task 函数）默认是多线程的！许多运行时使用一种称为 工作窃取 的方法，根据线程当前的使用情况透明地在不同线程之间移动任务，以提高系统的整体性能。这种方法实际上需要线程 和 任务，因此也需要未来值。

在考虑何时使用哪种方法时，请考虑以下经验法则：

  • 如果工作是非常并行的（即，CPU 密集型），例如处理可以分别处理的大量数据，线程是更好的选择。
  • 如果工作是非常并发的（即，I/O密集型），例如处理来自多个不同来源的消息，这些消息可能以不同的间隔或不同的速率到达，那么异步是一个更好的选择。

而且，如果您需要并行性和并发性，您不必在线程和异步之间做出选择。您可以自由地将它们一起使用，让每个都发挥其最擅长的作用。例如，列表 17-25 展示了这种混合在实际 Rust 代码中相当常见的一种情况。

摘要

这不会是你在这本书中最后一次看到并发。在第 21 章的项目中，将会在比这里讨论的更现实的情况下应用这些概念，并更直接地比较使用线程与任务和未来的解决问题方法。

无论你选择哪种方法，Rust 都为你提供了编写安全、快速、并发代码所需的工具——无论是用于高吞吐量的 Web 服务器还是嵌入式操作系统。

接下来，我们将讨论随着您的 Rust 程序变得更大，如何以惯用的方式建模问题和结构化解决方案。此外，我们还将讨论 Rust 的惯用法与您可能熟悉的面向对象编程的惯用法之间的关系。
